From Imperative to Declarative

NEAR Intents abstracts away complexity. This technical deep-dive explores the core components, cryptographic primitives, and on-chain logic that power a new era of Web3 transactions.

The Intent Data Structure

At its core, an intent is a structured data object that defines a user's desired outcome. Unlike a traditional transaction that specifies a function call and parameters, an intent is a declarative request that an off-chain network competes to fulfill.

Traditional Transaction

A low-level, imperative call to a specific smart contract function. It leaves no room for optimization and is publicly broadcasted to a mempool.

{
  "sender": "0xUserAddress",
  "to": "0xSwapContract",
  "value": "0",
  "data": "0x2e1a7d4d...",
  "gasLimit": "21000",
  "gasPrice": "50000000000"
}

NEAR Intent Object

A high-level, declarative request. Solvers compete to create the most optimal transaction bundle that achieves this outcome.

{
  "user": "alice.near",
  "goal": {
    "type": "swap",
    "from_token": "wrap.near",
    "to_token": "usdt.e.near",
    "amount": "100",
    "min_return_usd": "99.5"
  },
  "constraints": {
    "max_cost_usd": "0.10",
    "slippage_bps": "50"
  }
}

The Declarative Transaction Flow

This section details the 6-step lifecycle of an intent. Click on each step to reveal the technical actions taken by the user, solvers, and the on-chain verifier contract.

1

Intent Creation & Broadcast

▾

A user's client-side application generates a structured intent object. This object is then broadcasted to a private, off-chain network of solvers via a dedicated message passing system (e.g., a libp2p network or a specialized pub/sub channel).

2

Solver Bidding & Quote Generation

▾

Upon receiving the intent, solvers (AI or market makers) use their proprietary algorithms to find the most efficient path to fulfill it. This path may involve multiple steps across different chains and protocols. Each solver generates a signed "quote" that specifies the execution plan and the final guaranteed output for the user.

3

Quote Selection & User Commitment

▾

The user's client receives multiple quotes from various solvers. It can automatically select the best quote based on criteria like final token amount, latency, or solver reputation. The user then signs this quote, creating a cryptographic commitment.

4

On-Chain Submission to Verifier

▾

The chosen solver submits the signed quote and the necessary execution data to the on-chain Verifier smart contract. This submission is a single transaction that triggers the atomic settlement process.

5

Atomic Execution & Verification

▾

The verifier contract checks the cryptographic signatures of both the user and the solver. It then executes the series of sub-transactions (as defined by the solver's quote) atomically. If any part of the execution fails, the entire transaction is reverted, ensuring no partial state changes.

6

Settlement & Finalization

▾

After successful execution, the user's funds are moved, and the solver receives their fee. If the solver fails to fulfill their signed quote, a dispute can be initiated against them, leading to a penalty defined in the protocol.

The Role of Cryptography

NEAR Intents relies on robust cryptographic mechanisms to ensure security and trust without centralized control. Two key concepts are cryptographic commitments and session keys.

Cryptographic Commitments

When a user or a solver signs an intent or a quote, they are making a verifiable on-chain promise. This signature binds them to their stated intentions, and any deviation can be proven on-chain, leading to penalties.

// User signs the intent
const userSignature = sign(userKey, intentObject);

// Solver signs their quote
const solverSignature = sign(solverKey, quoteObject);

// Verifier contract checks these signatures
require(is_valid_signature(intent, user_signature, intent.user));
require(is_valid_signature(solver_quote, solver_signature, solver_quote.solver));

Session Keys

Users can grant limited, time-bound permissions to solvers through session keys. This allows solvers to act on the user's behalf for a specific intent without requiring the user's private key, greatly enhancing security and usability for complex transactions.

// User creates a temporary session key
const sessionKey = generateSessionKey();

// User signs a transaction to add session key to their account
add_key(sessionKey, {
    permission: {
        method_names: ["execute_intent"],
        receiver_id: "verifier.near"
    },
    duration: 3600 // expires in 1 hour
});

Solvers, Competition, and MEV Protection

Solvers are the core engine of the system. They operate in a private, off-chain environment, which is the primary reason intents are protected from typical MEV attacks like front-running.

Off-Chain Private Bidding

Unlike public mempools where transactions are visible to everyone, intents are broadcasted to a select, permissioned network of solvers. Solvers receive these intents privately and compete to offer the best price and execution. This prevents malicious actors from observing pending transactions and creating their own to front-run the user.

Universal Liquidity Aggregation

A solver's primary task is to find the most efficient path. This often involves aggregating liquidity from various sources: on-chain liquidity pools (DEXs), centralized exchanges (CeFi), and private over-the-counter (OTC) deals. The goal is to minimize slippage and maximize the final output for the user.

Gas Fee Abstraction

Solvers are not just providing a route; they are providing a service. They may absorb the underlying transaction fees (gas) as part of their business model, which is factored into their final quote. This provides a "gasless" experience for the user.

This chart illustrates the difference in price received for a large trade when leveraging different liquidity sources. A single DEX is often inefficient, while a DEX aggregator improves the price by pooling multiple on-chain sources. The NEAR Intents model, however, offers the best outcome by accessing a universal pool of DeFi, CeFi, and private OTC liquidity.

The On-Chain Verifier Contract

The verifier contract is the ultimate source of trust. It is an immutable smart contract that acts as a secure settlement layer, ensuring that a solver's off-chain promises are fulfilled on-chain in an atomic, non-custodial manner.

Verifier Logic (Pseudo-code)

The core function of the verifier contract is to validate a solver's proposal against a user's signed intent, and then execute it in a single, secure transaction.

// Verifier.near smart contract
fn execute_intent(intent, user_signature, solver_quote, solver_signature) {
    // 1. Verify user's signature on the intent
    require(is_valid_signature(intent, user_signature, intent.user));
    
    // 2. Verify solver's signature on the quote
    require(is_valid_signature(solver_quote, solver_signature, solver_quote.solver));
    
    // 3. Ensure the quote fulfills the intent's constraints
    // (e.g., minimum return amount, max slippage)
    require(solver_quote.min_return >= intent.goal.min_return_usd);
    
    // 4. Perform atomic execution
    // The verifier contract holds the user's funds temporarily via a session key.
    try {
        let funds_transferred = transfer(intent.goal.from_token, solver_quote.amount);
        let result = execute_solver_actions(solver_quote.actions);
        let final_amount = receive(intent.goal.to_token);
        
        // 5. Check post-conditions
        require(final_amount >= intent.goal.min_return_usd);
        
        // 6. Settle with user and solver
        transfer(final_amount, intent.user);
        transfer(solver_quote.fee, solver_quote.solver);
    } catch (e) {
        // If anything fails, revert all state changes.
        revert(e); 
    }
}
